## 一、正面评价

在列出问题之前，先肯定支付系统中做得好的方面：

- ✅ **幂等性设计优秀** — `activateOrder()` 使用原子状态守卫 `UPDATE ... WHERE status = 'pending'` 阻止重复激活，claim→execute→rollback 的三阶段模式正确
- ✅ **Alipay notify 签名验证** — 使用 `checkNotifySignV2`（RSA2），拒绝降级到 v1，防止伪造回调
- ✅ **provider 模式清晰** — `PaymentProvider` 接口抽象良好，支付宝/PayPal/微信统一接口，扩展方便
- ✅ **orderId 唯一性** — `ORDER_${orderId}_${Date.now()}` 保证每次支付生成的 trade_no 全局唯一
- ✅ **前端防重复提交** — `creatingRef` + `payingRef` 双重 ref 守卫，`processedReturnRef` 防止 return URL 重复处理
- ✅ **自适应轮询** — 前端轮询从 4s 起步，每次 +2s，上限 30s，共 45 次，兼顾响应速度和服务器压力
- ✅ **并发激活处理** — verify 端点对 `completing` 状态返回 202 + `retryAfter`，避免客户端重复提交
- ✅ **失败回滚** — `activateOrder` 中 catch 块将 `completing` 回滚为 `pending`，允许重试
- ✅ **notify 端点响应 "success"/"fail"** — 符合支付宝异步通知规范，失败时触发重推（最多 8 次）

## 🔴 严重问题

### #P1 payment_id 被覆盖导致旧 QR 码支付后无法匹配订单

- **严重程度**: 🔴 严重
- **文件**: `server/src/routes/payment.ts:232` + `server/src/routes/payment.ts:473`
- **问题描述**: 当用户在同一次结账流程中多次点击"支付"时，每次调用 `POST /orders/:id/pay` 都会执行 `UPDATE orders SET payment_id = ? WHERE id = ?`（第 232 行），**覆盖**之前的 `payment_id`。

**攻击/触发链路**:

```
1. 用户选择商品 → 创建订单 #1 → payment_id = "ORDER_1_1720000001"
2. 用户看到 QR 码 A（含 ORDER_1_1720000001）
3. 用户点击"返回"，不支付 QR 码 A
4. 用户再次点击"支付" → 订单 #1 的 payment_id 被覆盖为 "ORDER_1_1720000060"
5. 用户看到 QR 码 B（含 ORDER_1_1720000060）
6. 用户用支付宝扫了 QR 码 A（旧的）→ 支付成功
7. Alipay notify 到达，查询 `payment_id = 'ORDER_1_1720000001'`（第 473 行）
8. ❌ 找不到订单 — payment_id 已被覆盖为 ORDER_1_1720000060
9. notify 返回 'success'（第 478 行：阻止支付宝重试）
10. 💸 用户付了钱，但订单永远不会被激活
```

**更危险的变体**: 如果用户支付了**两个** QR 码（先扫旧的再扫新的），两个 notify 都会匹配到各自的 payment_id——第一个找不到（已被覆盖），第二个找到并激活。用户付了两次钱，只获得一次权益。

**当前条件**: `POST /orders` 创建订单时不检查已有 pending 订单，用户可以创建多个同一产品的 pending 订单（不同的 orderId），进一步放大此问题。

**修复建议**:

```typescript
// payment.ts:207-244 — 在 /orders/:id/pay 中
// 方案 A（最小改动）: 不覆盖已有 payment_id，改为创建新订单

router.post('/orders/:id/pay', authMiddleware, async (req, res) => {
  const orderId = Number(req.params.id);
  const order = await dbGet<OrderRow>(
    'SELECT * FROM orders WHERE id = ? AND user_id = ?',
    [orderId, req.userId]
  );
  if (!order) { res.status(404).json({ error: '订单不存在' }); return; }
  if (order.status !== 'pending') { res.status(400).json({ error: '订单已处理' }); return; }

  // 🔧 如果已有 payment_id，说明之前已发起过支付，创建一个新订单
  if (order.payment_id) {
    // 创建新订单（复用原订单信息）
    const newOrder = await dbRun(
      `INSERT INTO orders (user_id, plan_type, amount, currency, total_cents, payment_provider, status, coupon_code, metadata)
       VALUES (?, ?, ?, 'CNY', ?, ?, 'pending', ?, ?)`,
      [order.user_id, order.plan_type, order.amount, order.total_cents, order.payment_provider, order.coupon_code, order.metadata]
    );
    // 使用新订单 ID 继续
    // 将旧订单标记为 cancelled
    await dbRun("UPDATE orders SET status = 'cancelled' WHERE id = ?", [orderId]);
    // 返回新订单信息，前端用新 orderId 发起支付
    res.json({ success: true, data: { newOrderId: newOrder.lastInsertRowid, ... } });
    return;
  }

  // ... 原有支付发起逻辑
});
```

```typescript
// 方案 B（更彻底）: 在 POST /orders 创建时检查已有 pending 订单

// 在第 177 行 INSERT 之前添加:
const existingPending = await dbGet(
  "SELECT id FROM orders WHERE user_id = ? AND plan_type = ? AND status = 'pending'",
  [userId, planType]
);
if (existingPending) {
  // 返回已有订单，避免创建重复订单
  res.json({ success: true, data: { orderId: existingPending.id, ... } });
  return;
}
```

**建议同时实施方案 A + B**，彻底杜绝 payment_id 覆盖和重复订单。

## 🟠 高优先级

### #P2 `dbBatch` 不是真事务 — activateOrder 中途崩溃可能数据不一致

- **严重程度**: 🟠 高
- **文件**: `server/src/routes/payment.ts:341` + `server/src/models/database.ts:43-50`
- **问题描述**: 与 `code-audit-2026-7-12.md` #3 相同。`activateOrder()` 中的 `dbBatch(stmts)` 不是真正的事务——libsql 的 `batch()` 只是将多条 SQL 在一次 HTTP 往返中发送，但每条独立提交。中途崩溃可能导致：订阅已创建但优惠券未扣、积分已加但订单状态未更新等不一致状态。

**当前代码**:
```typescript
// payment.ts:341
await dbBatch(stmts); // ❌ 不是真事务
```

**修复建议**: 将 `activateOrder` 的核心 SQL 包装在 `BEGIN`/`COMMIT` 中：

```typescript
// 替换 dbBatch 调用
await dbRun('BEGIN');
try {
  for (const stmt of stmts) {
    await dbRun(stmt.sql, ...(stmt.args || []));
  }
  await dbRun('COMMIT');
} catch (err) {
  await dbRun('ROLLBACK');
  throw err;
}
```

### #P3 `orders.payment_id` 列缺少索引 — notify 端点每次都全表扫描

- **严重程度**: 🟠 高
- **文件**: `server/src/models/database.ts:111-126` (orders 表) + `server/src/routes/payment.ts:473`
- **问题描述**: Alipay notify 回调（第 473 行）通过 `WHERE payment_id = ?` 查询订单。如果 `orders` 表积累到数千行，每次 notify 都是一次全表扫描。notify 必须在 10 秒内响应 "success"，索引可以保证查询始终毫秒级。

**修复**: 在 `database.ts` 的迁移中添加：

```sql
CREATE INDEX IF NOT EXISTS idx_orders_payment_id ON orders(payment_id);
```

### #P4 缺少 rate limiting — 支付端点无频率限制

- **严重程度**: 🟠 高
- **文件**: `server/src/routes/payment.ts` (所有端点)
- **问题描述**: `POST /orders`、`POST /orders/:id/pay`、`POST /orders/:id/verify` 均无速率限制。恶意用户可以：

## 🟡 中等优先级

### #P5 订阅升级扣减逻辑可能导致极低价格升级

- **严重程度**: 🟡 中
- **文件**: `server/src/routes/payment.ts:127-151`
- **问题描述**: 月付转年付升级时，扣减金额 = `paidMonthly`（月付实付金额，上限为月付原价）。如果用户用大额优惠券购买了月付（实付 1 分钱），升级时可以扣减几乎全部的月付原价。

**当前逻辑**:
```typescript
const paidMonthly = Math.min(rawPaid, activeSub.price_cents); // 上限月付原价
totalCents = Math.max(100, totalCents - paidMonthly);        // 最低 1 元
```

**分析**: 
- `rawPaid` 来自 `lastMonthlyOrder.total_cents`——这是用户实付金额
- 如果用户用 90% 优惠券买月付，实付很低，但 `paidMonthly` 被限制为 `activeSub.price_cents`（月付标价）
- `activeSub.price_cents` 也来自 `products` 表，这是正确的定价基准
- 最低收费 100 分（1 元）是合理的底线

**结论**: 实际上是安全的——扣减上限是月付标价而非实付金额。但建议添加注释说明这个安全边界。

### #P7 用户可同时持有多个 pending 订单 — 可导致双重支付

- **严重程度**: 🟡 中
- **文件**: `server/src/routes/payment.ts:79-204`
- **问题描述**: `POST /orders` 不检查用户是否已有同一产品的 pending 订单。用户可以通过多个浏览器标签页创建多个 pending 订单，每个都生成不同的 QR 码。如果用户支付了其中两个，两个 notify 都会成功激活——用户付了两次钱获得两次权益。

**修复**: 与 #P1 的方案 B 合并——在创建订单前检查已有 pending 订单。

### #P8 前端 handleContinue 不创建订单 — 用户体验困惑

- **严重程度**: 🟡 中
- **文件**: `client/src/pages/CheckoutPage.tsx:237-249`
- **问题描述**: `handleContinue()` 从 review 步骤切换到 payment 步骤时**不创建订单**。订单在实际点击"支付"时（`handlePay`）才创建。这意味着：
  - 用户看到"支付"页面时，服务器上还没有订单
  - 如果用户在支付步骤停留很久（比如去拿手机），没有任何超时保护
  - review 步骤中的价格估算可能与实际订单价格不一致（如果有并发价格变动）

**建议**: 在 `handleContinue` 中就创建订单（状态为 `pending`），支付步骤只做 `POST /orders/:id/pay`。这样：
- 订单在支付前已存在，可以展示最终价格
- 可以在订单上设置过期时间

## 🔵 低优先级 / 建议

### #P9 `orders.amount` 列类型为 REAL — 浮点精度风险

- **严重程度**: 🔵 低
- **文件**: `server/src/models/database.ts:115`
- **问题描述**: `amount REAL NOT NULL` 存储浮点数金额。虽然代码中同时有 `total_cents INTEGER` 作为精确金额，但 `amount` 仍可能被直接读取使用（如 `admin/orders.ts:40` 的 fallback `Math.round(o.amount * 100)`）。
- **建议**: 长期应移除 `amount` 列，统一使用 `total_cents`。

### #P10 `getClient()` 在每次支付/查询时重复创建 AlipaySdk 实例

- **严重程度**: 🔵 低
- **文件**: `server/src/services/payment/alipay.ts:4-25`
- **问题描述**: `createPayment` 和 `verifyPayment` 每次都调用 `getClient()` 创建新的 `AlipaySdk` 实例。实例化包括 RSA 密钥解析，虽不昂贵但可避免。
- **建议**: 使用模块级单例懒加载：

```typescript
let _client: AlipaySdk | null = null;
function getClient(): AlipaySdk {
  if (_client) return _client;
  _client = new AlipaySdk({ ... });
  return _client;
}
```

### #P11 前端轮询无退避上限提示

- **严重程度**: 🔵 低
- **文件**: `client/src/pages/CheckoutPage.tsx:344`
- **问题描述**: 轮询最长可持续 `4+6+8+...+64 ≈ 45` 次 × 递增延迟，用户可能在 QR 码过期（15h）前一直在等。当前 45 次上限 × 30s 最大间隔 ≈ 最长约 15 分钟。虽合理，但用户看不到进度。
- **建议**: 显示轮询次数或预计剩余时间，如`"已等待 2 分钟，请确保已完成支付"`。

## 📋 支付链路架构图

```
用户选择商品                    Alipay 服务器
    │                              │
    ▼                              │
POST /orders ──→ 创建订单(pending)  │
    │                              │
    ▼                              │
POST /orders/:id/pay ──→ alipay.trade.precreate
    │                    生成 QR 码 + outTradeNo
    │                    更新 orders.payment_id
    │                              │
    ▼                              │
前端展示 QR 码 ←────────────────────┘
    │
    │  (用户扫码支付)
    │
    ├──→ 前端轮询 POST /orders/:id/verify
    │    └──→ alipay.trade.query
    │         └──→ TRADE_SUCCESS? → activateOrder()
    │
    └──→ (或) Alipay notify POST /alipay/notify
         └──→ 签名验证 → activateOrder()

activateOrder():
  1. 原子 CAS: pending → completing
  2. 更新 subscription / 加积分
  3. 消耗优惠券
  4. completing → completed
  5. 失败: completing → pending (回滚)
```
